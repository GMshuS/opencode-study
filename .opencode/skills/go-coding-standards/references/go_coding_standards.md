# Go 语言项目编码规范指南

## 一、 核心设计哲学
Go 语言以简洁、高效和一致性著称。在编写代码时，应始终遵循以下核心理念：
- **少即是多**：避免过度设计和复杂的抽象，保持代码直观易读。
- **显式优于隐式**：尽量减少隐藏行为，错误处理必须明确。
- **通信而非共享内存**：优先使用 Channel 进行并发控制，而非直接共享变量。

## 二、 项目目录结构规范
合理的项目结构是降低代码耦合度、提升团队协作效率的基础。推荐采用以下标准目录布局：

- **`App/`**：存放项目核心源代码，建议按业务模块进一步划分子目录。
- **`clib/`**：存放CGO依赖的C库的目录，建议按库名进一步划分子目录。
- **`docs/`**：存放项目设计文档、API文档、架构图及编译运行指南。
- **`internal/`**：内部共享包（私有包）。
- **`pkg/`**：公共工具包（可复用）。

## 三、 命名规范
清晰的命名是 Go 代码可读性的基石，严禁使用拼音或无意义的缩写。

1. **包名 (Package Names)**：必须简短、全小写且为单词（如 `user`, `httputil`）。禁止使用下划线或混合大小写，也无需在函数调用中重复包名。
2. **导出与非导出标识符**：
   - **首字母大写**：表示对外公开（Exported），如 `GetUser()`, `MaxRetries`。
   - **首字母小写**：表示包内私有（Unexported），如 `internalCache`, `maxConn`。
3. **变量与函数**：采用驼峰命名法（CamelCase/MixedCaps），不使用下划线。
   - 局部变量应尽量简短（如用 `count` 代替 `cnt`）。
   - 布尔类型变量推荐使用 `Is`, `Has`, `Can`, `Allow` 等前缀（如 `isAvailable`）。
4. **接口命名**：通常以 `I` 为前缀（如 `IReader`, `IWriter`）。
5. **struct命名**：通常以 `T` 为前缀（如 `TUser`, `TStudent`）。
6. **常量命名**：若为全局可见常量，采用大驼峰（如 `MaxBufferSize`）；若是包内私有常量，可采用小驼峰或全大写加下划线（如 `default_timeout`）。

## 四、 注释规范
Go 的注释不仅是给人看的，更是给 `godoc` 工具生成文档用的。统一使用中文注释，中英文字符及标点之间严格保留空格。

1. **包注释**：位于 `package` 声明之前，简述包的功能、用法及创建信息。
   ```go
   // database 包提供了与 MySQL 交互的核心函数和工具。
   // 创建人：DevTeam
   // 创建时间：20240520
   package database
   ```
2. **结构与接口注释**：放在定义的前一行，说明其用途。成员变量注释需对齐并置于其后。
   ```go
   // User, 表示系统中的用户实体对象
   type TUser struct {
       ID       int64  // 用户唯一标识
       UserName string // 用户名
   }
   ```
3. **函数/方法注释**：必须以函数名开头，使用逗号分隔说明部分。包含参数列表和返回值说明。
   ```go
   // GetUserByID, 根据指定的ID查询用户详细信息。
   // id: 用户的唯一数字标识
   // err: 若未找到返回 ErrUserNotFound，其他情况返回底层错误
   func GetUserByID(id int64) (*User, error) { ... }
   ```
4. **逻辑注释**：仅在复杂算法或非直觉的业务妥协处添加，解释“为什么这么做”而非“做了什么”。

## 五、 导入包规范 (Imports)
标准化的导入顺序有助于快速定位依赖来源，不同分组之间必须使用空行分隔：

1. **标准库**：如 `context`, `fmt`, `log`。
2. **第三方库**：如 `github.com/gin-gonic/gin`, `go.uber.org/zap`。
3. **项目内部包**：如 `your-project/internal/models`。

*强制要求*：强烈建议使用 `goimports` 工具自动管理导入，移除未使用的包并补全缺失的包。禁止在项目中通过相对路径引入非本项目的包。

## 六、 错误处理规范
Go 语言没有异常机制，错误是一等公民，必须被妥善处理。

1. **绝不丢弃错误**：禁止使用 `_` 忽略可能产生错误的返回值。接收到错误后，必须返回给上层调用者或使用日志记录。
2. **错误包装与上下文**：使用 `fmt.Errorf("failed to get user: %w", err)` 包装错误，保留原始错误链以便排查。
3. **慎用 Panic**：除非遇到不可恢复的致命错误（如程序初始化失败），否则一律通过返回 `error` 处理业务异常。
4. **错误描述格式**：自定义的错误消息如果是英文，必须全小写且不以标点符号结尾。

## 七、 代码格式化与排版
Go 社区对代码风格有严格的共识，拒绝主观的格式争论。

1. **强制格式化**：所有代码提交前必须运行 `gofmt` 或 IDE 的等效格式化工具。
2. **缩进规则**：严格使用制表符（Tab）进行缩进，禁止使用空格。
3. **行长限制**：建议单行不超过 120 个字符。过长的参数列表应在左括号后换行，并对齐或额外缩进一个 Tab。
4. **大括号位置**：左大括号 `{` 必须与控制语句（`if`, `for`, `func`）在同一行，严禁另起一行（防止自动分号插入导致的 Bug）。
5. **日志规范**：日志函数的第一个参数通常为 `context.Context`。打印日志前必须对密码、密钥等敏感信息进行脱敏处理。

## 八、 并发与数据结构最佳实践
1. **通道优先**：使用 Channel 传递数据以实现 Goroutine 间的同步，尽量避免跨协程共享内存。
2. **安全启动**：启动 Goroutine 时必须明确其生命周期，防止 Goroutine 泄漏。在需要捕获 panic 的场景，必须在 Goroutine 内部使用 `defer recover()`。
3. **切片与映射初始化**：优先使用 `make` 初始化 Slice、Map 和 Channel。当预知容量时，务必传入 `cap` 参数以避免运行时频繁扩容。
4. **接收器选择**：如果方法需要修改结构体状态或结构体较大，使用指针接收器；否则使用值接收器以保持语义清晰。
