# Python 代码审查检查清单

## 架构设计审查

### 模块化设计
- [ ] **包结构**
  - 包职责是否单一（单一职责原则）
  - 目录结构是否清晰反映功能划分
  - `__init__.py` 是否合理使用
  - 是否避免循环导入

- [ ] **依赖管理**
  - 依赖关系是否单向（避免循环依赖）
  - 第三方依赖是否在 `requirements.txt` 或 `pyproject.toml` 中声明
  - 依赖版本是否锁定（使用 `==` 或约束文件）
  - 是否使用虚拟环境（venv/conda/poetry）

- [ ] **导入组织**
  - 导入顺序是否规范（标准库→第三方→本地）
  - 是否避免 `from module import *`
  - 导入是否放在文件顶部
  - 是否使用绝对导入优先

### 接口设计
- [ ] **API 设计**
  - 函数签名是否清晰
  - 参数是否有类型注解
  - 返回值是否有类型注解
  - 是否有文档字符串（docstring）

- [ ] **抽象层次**
  - 是否使用抽象基类（`abc.ABC`）
  - 接口与实现是否分离
  - 是否遵循依赖倒置原则

## 类型安全审查

### 类型注解
- [ ] **类型提示**
  - 函数参数是否有类型注解
  - 返回值是否有类型注解
  - 变量是否需要类型注解（复杂类型）
  - 是否使用 `Optional`/`Union`/`Any` 恰当

- [ ] **类型检查**
  - 是否使用 `mypy` 或类似工具检查
  - 类型注解与实际使用是否一致
  - 泛型使用是否正确（`List[T]`, `Dict[K, V]`）

## 错误处理审查

### 异常处理
- [ ] **异常捕获**
  - 是否避免裸 `except:` 子句
  - 是否捕获特定异常类型
  - 异常是否记录日志
  - 异常链是否正确维护（`raise ... from ...`）

- [ ] **异常抛出**
  - 是否使用内置异常类型
  - 自定义异常是否继承 `Exception`
  - 异常信息是否清晰有用
  - 是否在适当层级处理异常

- [ ] **资源清理**
  - 是否使用 `with` 语句管理资源
  - `finally` 块是否正确清理资源
  - 上下文管理器是否实现（`__enter__`/`__exit__`）

### 防御性编程
- [ ] **输入验证**
  - 函数参数是否验证
  - 边界条件是否检查
  - `None` 值是否处理
  - 空集合是否处理

- [ ] **断言使用**
  - `assert` 是否用于调试（非错误处理）
  - 断言条件是否合理
  - 是否避免副作用在断言中

## 内存管理审查

### 对象生命周期
- [ ] **引用管理**
  - 是否避免循环引用
  - 是否使用 `weakref` 打破循环引用
  - 大对象是否及时释放

- [ ] **生成器与迭代器**
  - 大数据集是否使用生成器
  - 迭代器是否正确实现（`__iter__`/`__next__`）
  - 是否使用 `yield` 简化迭代器

### 容器使用
- [ ] **列表/字典**
  - 列表推导式是否替代 `map`/`filter`
  - 字典推导式是否替代循环构建
  - 预分配是否考虑（已知大小）
  - 是否在遍历中修改容器

## 并发安全审查

### 线程安全
- [ ] **GIL 考虑**
  - CPU 密集型是否使用 `multiprocessing`
  - I/O 密集型是否使用 `threading`/`asyncio`
  - 共享状态是否加锁保护

- [ ] **锁机制**
  - `threading.Lock`/`RLock` 是否正确使用
  - 是否使用 `with lock:` 语法
  - 是否避免死锁（锁顺序一致）
  - `threading.Condition` 是否正确使用

### 异步编程
- [ ] **Async/Await**
  - 是否使用 `asyncio` 管理异步操作
  - `async`/`await` 使用是否正确
  - 协程是否被正确调度
  - 是否避免阻塞异步代码

- [ ] **任务管理**
  - `asyncio.Task` 是否正确创建和等待
  - 超时是否设置
  - 取消是否正确处理
  - 异常是否在任务中捕获

### 进程间通信
- [ ] **多进程**
  - `multiprocessing` 是否正确使用
  - 进程间通信是否安全（`Queue`/`Pipe`）
  - 共享内存是否正确使用
  - 进程池是否正确关闭

## 资源管理审查

### 文件操作
- [ ] **文件处理**
  - 文件是否使用 `with` 语句打开
  - 文件编码是否指定
  - 大文件是否流式处理
  - 临时文件是否清理（`tempfile`）

### 网络连接
- [ ] **网络资源**
  - Socket 连接是否正确关闭
  - HTTP 连接是否使用连接池
  - 请求超时是否设置
  - 重试机制是否实现

### 数据库
- [ ] **数据库连接**
  - 连接是否正确关闭
  - 连接池是否使用
  - 事务是否正确提交/回滚
  - 是否防止 SQL 注入（参数化查询）

## 代码质量审查

### 命名规范
- [ ] **PEP 8 命名**
  - 模块名是否小写（可带下划线）
  - 类名是否大驼峰（`ClassName`）
  - 函数名是否小写带下划线（`function_name`）
  - 常量是否全大写（`CONSTANT_NAME`）
  - 私有成员是否单下划线前缀（`_private`）
  - 名称修饰是否双下划线前缀（`__mangled`）

- [ ] **命名语义**
  - 变量名是否有意义
  - 布尔变量是否使用 `is_`/`has_` 前缀
  - 避免缩写（除非通用）

### 代码风格
- [ ] **格式化**
  - 是否使用 `black` 或 `autopep8` 格式化
  - 行宽是否合理（PEP 8 建议 79/88 字符）
  - 缩进是否 4 空格（非 Tab）
  - 空行是否合理使用

- [ ] **文档字符串**
  - 模块是否有文档字符串
  - 类是否有文档字符串
  - 函数是否有文档字符串
  - 是否遵循 docstring 约定（Google/NumPy/Sphinx）

- [ ] **注释**
  - 注释是否解释"为什么"而非"做什么"
  - 是否避免冗余注释
  - `TODO`/`FIXME`/`XXX` 是否标注
  - 中文注释是否使用（项目约定）

- [ ] **函数设计**
  - 函数长度是否合理（建议<50 行）
  - 参数个数是否合理（建议≤5 个）
  - 是否使用默认参数
  - 是否使用 `*args`/`**kwargs` 恰当

### 魔法值
- [ ] **常量提取**
  - 魔法数字是否提取为具名常量
  - 配置值是否可配置
  - 是否使用 `enum.Enum` 定义枚举

  - 字典查找是否替代线性搜索
  - 集合操作是否使用（`set`）
  - `defaultdict`/`Counter` 是否使用恰当

- [ ] **惰性求值**
  - 是否使用生成器替代列表

  - `lazy` 加载是否实现

### 缓存与记忆化

  - 内存泄漏是否避免

### 代码覆盖率

- [ ] **默认参数陷阱**

  - 是否使用 `None` + 内部初始化

### 装饰器

## 常见 Python 陷阱

| 陷阱 | 说明 | 检查点 |
|------|------|--------|
| 可变默认参数 | 默认参数在函数定义时求值 | `def f(x=[])` |
| 闭包变量捕获 | 循环变量在闭包中被共享 | `for` 循环中的 `lambda` |
| 类变量 vs 实例变量 | 类变量被所有实例共享 | `class A: x = 1` |
| LEGB 规则 | 变量作用域查找顺序 | Local→Enclosing→Global→Builtins |
| 浅拷贝 vs 深拷贝 | 嵌套对象拷贝问题 | `copy()` vs `deepcopy()` |
| 迭代器耗尽 | 生成器只能遍历一次 | `next()` 后再次遍历 |
| 浮点数精度 | 浮点数比较问题 | `0.1 + 0.2 != 0.3` |
| 字符串驻留 | 字符串身份比较问题 | `is` vs `==` |
| 模块级单例 | 模块导入即单例 | 依赖模块级变量 |
| GIL 限制 | 多线程无法利用多核 | CPU 密集型任务 |

## 审查报告模板要点

2. **优点** - 先肯定做得好的地方
3. **问题分级** - Critical/Major/Minor/Potential
4. **代码统计** - 行数、函数数、测试覆盖率
5. **修复优先级** - 高/中/低
6. **代码对比** - 修复前后的具体代码
