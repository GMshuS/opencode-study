# JavaScript/TypeScript 代码审查检查清单

## 架构设计审查

### 模块化设计
- [ ] **模块组织**
  - 模块职责是否单一（单一职责原则）
  - 目录结构是否清晰反映功能划分
  - 是否使用 ES6 模块系统（`import`/`export`）
  - 是否避免循环依赖

- [ ] **依赖管理**
  - 依赖关系是否单向（避免循环依赖）
  - 第三方依赖是否在 `package.json` 中声明
  - 依赖版本是否锁定（`package-lock.json`/`yarn.lock`）
  - 是否区分 `dependencies` 和 `devDependencies`

- [ ] **导入组织**
  - 导入顺序是否规范（第三方→本地→样式）
  - 是否避免默认导出优先（具名导出优先）
  - 是否使用路径别名（`tsconfig.json` paths）
  - 是否避免深层嵌套导入

### 接口设计（TypeScript）
- [ ] **类型定义**
  - 接口是否使用 `interface` 或 `type` 定义
  - 类型是否 exported（需要时）
  - 泛型使用是否合理
  - 是否使用联合类型/交叉类型恰当

- [ ] **API 设计**
  - 函数签名是否清晰
  - 参数是否有类型注解
  - 返回值是否有类型注解
  - 是否有 JSDoc 注释

## 类型安全审查（TypeScript）

### 类型注解
- [ ] **类型提示**
  - 函数参数是否有类型注解
  - 返回值是否有类型注解
  - 变量是否需要类型注解（类型推断不足时）
  - 是否避免使用 `any`

- [ ] **严格模式**
  - 是否启用 `strict` 模式
  - `noImplicitAny` 是否启用
  - `strictNullChecks` 是否启用
  - `noUnusedLocals`/`noUnusedParameters` 是否启用

- [ ] **类型守卫**
  - 是否使用类型守卫函数
  - `typeof`/`instanceof` 检查是否正确
  - 类型断言是否必要（`as Type`）
  - 非空断言是否安全（`!`）

## 错误处理审查

### 异常处理
- [ ] **Try/Catch**
  - 异步代码是否使用 `try/catch`
  - Promise 是否有 `.catch()` 处理
  - 是否避免空的 catch 块
  - 错误是否记录日志

- [ ] **错误类型**
  - 是否使用自定义错误类
  - 错误信息是否清晰有用
  - 错误码是否统一定义
  - 错误是否被正确传播

- [ ] **Promise 处理**
  - Promise 是否始终被处理（避免未处理拒绝）
  - `async/await` 是否有错误处理
  - `Promise.all` 是否有错误处理
  - 是否使用 `Promise.race` 设置超时

### 防御性编程
- [ ] **输入验证**
  - 函数参数是否验证
  - 边界条件是否检查
  - `null`/`undefined` 是否处理
  - 空数组/对象是否处理

- [ ] **可选链与空值合并**
  - 是否使用可选链（`?.`）安全访问
  - 是否使用空值合并（`??`）提供默认值
  - 是否避免 `||` 误判 falsy 值

## 内存管理审查

### 对象生命周期
- [ ] **引用管理**
  - 是否避免内存泄漏（事件监听器清理）
  - 定时器是否清除（`clearTimeout`/`clearInterval`）
  - 订阅是否取消（Observable/EventEmitter）
  - 大对象是否及时释放

- [ ] **闭包使用**
  - 闭包是否捕获必要变量
  - 是否避免闭包内存泄漏
  - 循环中的闭包是否正确处理

### 数据结构
- [ ] **数组/对象操作**
  - 数组操作是否使用不可变模式
  - 对象更新是否使用扩展运算符
  - 是否在遍历中修改集合
  - 大数组操作是否考虑性能

## 并发安全审查

### 异步编程
- [ ] **Promise/Async**
  - `async/await` 使用是否正确
  - 并行操作是否使用 `Promise.all`
  - 是否避免串行异步调用（可并行时）
  - 异步操作超时是否设置

- [ ] **竞态条件**
  - 是否有竞态条件风险
  - 请求取消是否处理（`AbortController`）
  - 防抖/节流是否使用恰当
  - 加载状态是否正确管理

### 事件处理
- [ ] **事件监听**
  - 事件监听器是否移除
  - 事件处理是否异步
  - 事件冒泡是否阻止（需要时）
  - 事件委托是否使用

### Web Workers
- [ ] **多线程**
  - CPU 密集型是否使用 Web Worker
  - Worker 通信是否正确（`postMessage`）
  - Worker 是否正确终止
  - SharedArrayBuffer 是否正确使用

## 资源管理审查

### 网络请求
- [ ] **HTTP 请求**
  - 请求是否设置超时
  - 重试机制是否实现
  - 请求取消是否支持
  - 响应数据是否验证

- [ ] **缓存策略**
  - 请求结果是否缓存（需要时）
  - 缓存失效是否处理
  - 本地存储是否使用恰当（LocalStorage/SessionStorage）
  - 敏感数据是否不存储

### DOM 操作
- [ ] **DOM 管理**
  - DOM 引用是否清理
  - 事件监听器是否移除
  - 内存泄漏是否避免（IE 兼容）
  - 虚拟 DOM 是否使用（框架中）

## 代码质量审查

### 命名规范
- [ ] **标识符命名**
  - 变量名是否小驼峰（`variableName`）
  - 类名是否大驼峰（`ClassName`）
  - 常量是否全大写（`CONSTANT_NAME`）
  - 私有成员是否 `#` 前缀（ES2022）
  - 布尔变量是否使用 `is`/`has`/`should` 前缀

- [ ] **文件命名**
  - 文件命名是否一致（kebab-case/camelCase）
  - 组件文件是否大写（React 组件）
  - 测试文件是否 `*.test.ts`/`*.spec.ts`

### 代码风格
- [ ] **格式化**
  - 是否使用 Prettier 格式化
  - 缩进是否一致（2 空格或 4 空格）
  - 行宽是否合理（80/100/120 字符）
  - 分号使用是否一致

- [ ] **注释**
  - 是否有 JSDoc 注释
  - 注释是否解释"为什么"而非"做什么"
  - `TODO`/`FIXME` 是否标注
  - TypeScript 是否有 `@param`/`@returns`

- [ ] **函数设计**
  - 函数长度是否合理（建议<50 行）
  - 参数个数是否合理（建议≤3 个，使用对象参数）
  - 是否使用默认参数
  - 是否使用剩余参数（`...args`）

### 代码模式
- [ ] **ES6+ 特性**
  - 是否使用 `const`/`let`（避免 `var`）
  - 是否使用箭头函数
  - 是否使用解构赋值
  - 是否使用模板字符串
  - 是否使用展开/收集运算符

- [ ] **函数式编程**
  - 是否使用 `map`/`filter`/`reduce`
  - 是否避免副作用
  - 是否使用纯函数
  - 是否使用不可变数据

  - 防抖/节流是否使用

### 缓存策略

## 安全审查

  - `eval()` 是否避免
  - `dangerouslySetInnerHTML` 是否谨慎使用

- [ ] **CSRF 防护**
  - CSRF Token 是否使用
  - SameSite Cookie 是否设置
  - 敏感操作是否验证

### 数据安全
- [ ] **敏感数据**

  - API 密钥是否不硬编码

### 依赖安全

## 常见 JavaScript/TypeScript 陷阱

| 陷阱 | 说明 | 检查点 |
|------|------|--------|
| 类型 coercion | 隐式类型转换 | `==` vs `===` |
| this 绑定 | `this` 指向问题 | 回调函数中的 `this` |
| 异步时序 | 异步执行顺序 | Promise 链/async-await |
| 闭包变量 | 循环中闭包捕获变量 | `for` 循环中的 `var` |
| 原型污染 | 修改 Object.prototype | 库代码安全 |
| 事件循环 | 宏任务/微任务顺序 | `setTimeout` vs `Promise` |
| 浮点精度 | IEEE 754 精度问题 | `0.1 + 0.2 !== 0.3` |
| 对象引用 | 对象按引用传递 | 修改原对象 |
| 数组陷阱 | `delete` vs `splice` | 数组长度变化 |
| 模块缓存 | ES 模块单例特性 | 模块级变量共享 |

## 审查报告模板要点

1. **项目概述** - 功能说明、目录结构
2. **优点** - 先肯定做得好的地方
3. **问题分级** - Critical/Major/Minor/Potential
4. **代码统计** - 行数、函数数、测试覆盖率
5. **修复优先级** - 高/中/低
6. **代码对比** - 修复前后的具体代码
