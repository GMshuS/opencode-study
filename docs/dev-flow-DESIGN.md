# dev-flow 全流程开发框架设计文档

## 1. 概述

dev-flow 是一套基于 Master-Agent 模式的 AI 驱动全流程开发框架，覆盖从需求探索到代码交付的完整软件开发生命周期。它由 6 个 agent 组成，其中 dev-brainstorm 作为前置预热阶段独立运行，dev-flow 作为总调度编排其余 4 个子 agent 完成开发闭环。

### 核心价值

- **全流程自动化**：brainstorm 探索 → 需求规划 → 分批编码 → 审查 → 修复回环 → 交付，减少人工介入
- **前置深度设计**：dev-brainstorm 在编码前完成 7 阶段方案设计（现有系统→5W1H→技术选型→架构→风险→路线图→交付），避免方向性返工
- **故障可恢复**：状态文件（原子写入）+ 会话统一管理，支持 crash 后恢复
- **质量闭环**：构建验证 + 代码审查 → 自动修复回环（最多 3 轮）
- **依赖驱动分批**：任务依赖图分层 + 线性链折叠优化，减少子 agent 调用轮次
- **分层验证**：dev-review 做完整质量门禁（构建+类型+Linter+审查+测试），dev-bugfix 做轻量编译自检
- **文件即契约**：子 agent 之间通过文件通信，dev-flow 不进行字段级解析

## 2. 架构设计

### 2.1 全貌：主从代理 + 前置预热

```
                        ┌────────────────────────┐
                        │    dev-brainstorm        │  ← Primary Agent（前置预热）
                        │    7阶段方案设计          │
                        │    交互式需求探索          │
                        └───────────┬──────────────┘
                                    │ 产出 brainstorm.md
                                    ▼
                        ┌──────────────────────────┐
                        │    dev-flow (dev-flow)    │  ← Primary Agent（总调度）
                        │    流程编排 / 状态管理      │
                        └──┬───┬───┬───┬──────────┘
                           │   │   │   │
                    ┌──────┘   │   │   └──────────┐
                    ▼          ▼   ▼              ▼
            ┌──────────┐ ┌────────┐ ┌────────┐ ┌──────────┐
            │ dev-plan │ │dev-code│ │dev-review│ │dev-bugfix│
            │ 只读规划   │ │ 编码    │ │质量门禁   │ │ 修复     │
            │subagent  │ │subagent│ │subagent  │ │subagent  │
            └──────────┘ └────────┘ └──────────┘ └──────────┘
```

### 2.2 组件角色矩阵

| 组件 | 模式 | 读写 | Bash | 温度 | 职责 |
|------|------|------|------|------|------|
| **dev-brainstorm** | primary | 读写 | 完整 | 0.7 | 新需求方案设计：现有系统分析→需求挖掘→技术选型→架构→风险→路线图→交付 |
| **dev-flow** | primary | 读写 | 完整 | 0.0 | 流程总调度，状态管理，subagent 编排，依赖分层与链折叠，交付汇总 |
| **dev-plan** | subagent | 只读 | 无 | 0.0 | 需求分析，技术选型，架构设计，任务拆分（含验收条件） |
| **dev-code** | subagent | 读写 | 完整 | 0.2 | 按计划分批编码，语法自检，更新 plan.md 任务勾选，依赖安装 |
| **dev-review** | subagent | 只读 | 完整 | 0.0 | 构建验证 + 类型检查 + Linter + 代码审查 + 测试验证 |
| **dev-bugfix** | subagent | 读写 | 受限 | 0.1 | BUG 定位与修复，编译自检，回归验证 |

### 2.3 核心设计原则

1. **plan.md 是真理来源（SSOT）**：所有子 agent 从 plan.md 获取语言/框架、架构方案、验收条件
2. **文件即契约**：各子 agent 的输出格式在其定义文件中约定，dev-flow 不进行字段级解析
3. **修改只影响一点**：如需增减字段，只需修改对应子 agent 的 agent 定义文件
4. **幂等性**：每个步骤可重复执行，状态文件确保流程连续性
5. **最小权限**：只读 agent（dev-plan, dev-review）没有 write/edit 权限
6. **人工兜底**：所有自动化路径都有上报人工介入的逃生口
7. **状态持久化**：JSON 状态文件 + git diff 双重保障
8. **前置预热**：dev-brainstorm 在 dev-flow 启动前完成深度方案设计，确保方向正确

## 3. 工作流详解

### 3.1 阶段 0：前置方案设计（dev-brainstorm，可选）

dev-brainstorm 是 dev-flow 的**前置可选阶段**，用于在进入正式开发前进行深度的需求探索和方案设计。它对**新需求/新功能**适用，不处理纯代码解读请求。

```
用户触发 /dev-brainstorm <项目名> <主题>
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│ 入口门控：请求分类                                         │
│  • 类型 A（方案设计）→ 进入流程                             │
│  • 类型 B（代码解读）→ 输出指引，退出                        │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 初始化 → 创建 .brainstorm-state.json                       │
│  SET $DOC_PATH = ./dev-flow/$PROJECT_NAME               │
└──────────────────────────┬───────────────────────────────┘
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
   │ 阶段1: 现有系统│ │ 阶段2: 5W1H  │ │ 阶段3: 技术选型│
   │ 理解          │ │ 需求挖掘     │ │ 分析          │
   │ (棕地项目)    │ │ (6维度提问)  │ │ (≥2方案对比)  │
   └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
          │                │                │
          ▼                ▼                ▼
   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
   │ 阶段4: 架构   │ │ 阶段5: 风险  │ │ 阶段6: 实施   │
   │ 方案探索      │ │ 评估         │ │ 路线图        │
   │ (2种候选架构) │ │ (四类风险)   │ │ (MVP→二→三)  │
   └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
          │                │                │
          └────────────────┼────────────────┘
                           ▼
                  ┌──────────────────┐
                  │ 阶段7: 方案交付    │
                  │ 写入 brainstorm.md │
                  │ 含 YAML frontmatter│
                  └────────┬─────────┘
                           │
                           ▼
                  启动 dev-flow 时传入 brainstorm.md 作为上下文
```

**各阶段详情**：

| 阶段 | 内容 | 输出 | 用户确认 |
|------|------|------|---------|
| 1 — 现有系统理解 | 阅读关键模块，整理项目结构、模块职责、相关机制 | 现有系统分析 | 必须 |
| 2 — 5W1H 需求挖掘 | What/Why/Who/Where/When/How 六维提问 | 需求分析表 | 必须 |
| 3 — 技术选型分析 | 每项技术决策 ≥2 方案，5维对比 | 技术决策对比表 | 逐项确认 |
| 4 — 架构方案探索 | 2 种候选架构，含架构图、分层、数据流 | 架构方案对比 | 必须 |
| 5 — 风险评估 | 技术/依赖/进度/安全四类风险矩阵 | 风险矩阵 | 必须 |
| 6 — 实施路线图 | MVP → 阶段二 → 阶段三，含完成标准 | 分阶段路线图 | 必须 |
| 7 — 方案交付 | 3 项最终确认 → 写入文件 → 验证 | brainstorm.md | 必须 |

**交付后提示**：
```
/dev-flow/dev-flow $PROJECT_NAME $DOC_PATH/brainstorm.md
```

### 3.2 阶段 1~4：dev-flow 总调度

```
用户触发 /dev-flow <功能名> [brainstorm.md路径]
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│ 初始化                                                    │
│  • 提取 $FEATURE_NAME                                     │
│  • 检查是否提供 brainstorm.md 路径（如有则传给 dev-plan）    │
│  • 原子创建 .flow-state.json（{iteration:0, status:"planning"}）│
│  • 校验状态文件 JSON 完整性                                 │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤1：探索需求、生成开发计划                                │
│  • 构建 prompt（原始需求 + brainstorm_path（如有））        │
│  • 调用 @dev-plan（只读规划）                              │
│  • 结果写入 plan.md                                      │
│  • 更新状态：planned                                     │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤1.2：计划确认（强制，≤ 5 轮）                           │
│  • 展示摘要：技术选型 / 架构 / 文件结构 / 任务清单 / 风险     │
│  • 提问：「以上开发计划是否确认？如需调整请说明」             │
│  ┌── 确认 ──► plan-confirmed                              │
│  ├── 修改 ──► plan-revising → 重新调用 dev-plan（≤5轮）     │
│  └── 终止 ──► cancelled                                   │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤2：按依赖分批编码（含链折叠优化）                         │
│  • 从 plan.md 解析任务依赖图，按 Level 分层                  │
│  • 线性链检测与折叠：扫描"单进单出"任务链（入度=1且出度=1）   │
│    - 链长 ≥ 2 → 折叠为同一批次                             │
│    - 批次名格式：链首任务→链尾任务                           │
│  • 依赖深度检查：总层级 > 3 → 输出告警（不阻断）             │
│  • 按折叠后批次顺序分批调用 @dev-code                       │
│  • 每批：plan.md → dev-code → 写入 code.md（追加）          │
│  • 批次完成判定：grep plan.md 核验本批任务是否全部 [x]       │
│  • 前一批未完成不启动下一批                                 │
│  • 全量终检：所有批次完成 → grep plan.md 全部 [x]            │
│  • 更新状态：coded                                        │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤3：审查与BUG修复（闭环，≤ 3 轮）                        │
│  • 调用 @dev-review（只读，完整质量门禁）                   │
│  • 结果写入 review.md                                     │
│  • 判断审查结论：                                          │
│    ├── 通过 → 进入步骤4                                    │
│    └── 不通过：                                            │
│        ├── iteration >= 3 → status: "failed"，上报终止     │
│        └── iteration < 3 → @dev-bugfix → 写入 bugfix.md   │
│            → iteration += 1 → 重新审查（回到本步骤开头）     │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤4：交付成果                                           │
│  • 从 code.md + bugfix.md 提取文件路径 → modified_files.txt │
│  • 生成 4 段式提交信息并展示                                │
│  • 提示用户可通过 @git-autocommit 提交                     │
│  • 更新状态：delivered                                    │
└──────────────────────────────────────────────────────────┘
```

### 3.3 状态机（dev-flow）

```
                    ┌──────────────────┐
                    │  planning (0)     │ ← 初始化
                    └────────┬─────────┘
                             │ dev-plan 完成
                             ▼
                    ┌──────────────────┐
                    │    planned        │
                    └────────┬─────────┘
                             │ 步骤1.2 用户确认
                  ┌──────────┼────────────┐
                  ▼          ▼            ▼
           ┌──────────┐ ┌──────────┐ ┌──────────┐
           │plan-conf.│ │plan-rev. │ │cancelled │
           │ 确认通过  │ │ 要求修改  │ │ 用户终止  │
           └────┬─────┘ └────┬─────┘ └──────────┘
                │            │
                │            └──→ 回退 planning
                ▼
           ┌──────────────────┐
           │ plan-confirmed   │
           └────────┬─────────┘
                    │ dev-code 完成（全量终检）
                    ▼
           ┌──────────────────┐
           │     coded         │
           └────────┬─────────┘
                    │ dev-review 完成
                    ▼
           ┌──────────────────┐
           │    reviewed       │
           └──┬───────────────┘
              │ 审查结论？
    ┌─────────┴─────────┐
    ▼                   ▼
┌──────────────┐   ┌──────────────┐
│  delivered   │   │   bugfixed   │ ← iteration += 1
└──────────────┘   └──────┬───────┘
                          │ 回退到 reviewed
                          ▼
                   ┌──────────────┐
                   │  reviewed    │ ← 重新审查
                   └──────┬───────┘
                          │ iteration >= 3 ?
                          ├── 否 → bugfixed
                          └── 是 → failed
```

### 3.4 状态说明

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| `planning` | 初始化完成，规划中 | dev-flow 启动 |
| `planned` | 规划完成，等待确认 | dev-plan 返回 |
| `plan-confirmed` | 用户确认计划 | 步骤1.2 用户确认 |
| `plan-revising` | 用户要求修改计划 | 步骤1.2 用户选择修改 |
| `cancelled` | 用户终止流程 | 步骤1.2 用户选择终止 |
| `coded` | 编码完成 | dev-code 全量终检通过 |
| `reviewed` | 审查完成 | dev-review 返回 |
| `bugfixed` | 修复完成（iteration+1） | dev-bugfix 返回 |
| `delivered` | 交付完成 | 步骤4 完成 |
| `failed` | 超限终止（iteration≥3） | 修复循环超限 |

## 4. 各 Agent 详述

### 4.1 dev-brainstorm（前置方案设计）

| 属性 | 值 |
|------|-----|
| 模式 | primary agent |
| 温度 | 0.7 |
| 权限 | 读写 + bash（完整）+ webfetch |
| 状态文件 | `.brainstorm-state.json` |
| 关键配置 | 只能修改 brainstorm.md 和状态文件 |

**入口门控**：

- **类型 A（方案设计）**：新需求/新功能的技术方案设计，进入 7 阶段流程
- **类型 B（代码解读）**：纯代码运行机制理解，输出指引后退出

**7 阶段流程**：

| 阶段 | current-phase | 核心动作 | 用户确认 |
|------|:---:|---------|:---:|
| 初始化 | — | 生成 `$DATE`，设置 `$DOC_PATH`，初始化状态文件 | — |
| 1 — 现有系统理解 | 1 | 阅读关键模块 → 整理项目结构+模块职责+相关机制 | 必须 |
| 2 — 5W1H 需求挖掘 | 2 | What/Why/Who/Where/When/How 六个维度提问并整理 | 必须 |
| 3 — 技术选型分析 | 3 | 每项技术决策 ≥2 方案，5 个维度对比 | 逐项确认 |
| 4 — 架构方案探索 | 4 | 2 种候选架构（含文字架构图、分层、数据流、对比表） | 必须 |
| 5 — 风险评估 | 5 | 四类风险矩阵（技术/依赖/进度/安全），严重程度+概率+应对 | 必须 |
| 6 — 实施路线图 | 6 | 分阶段实施计划（MVP→阶段二→阶段三），明确完成标准 | 必须 |
| 7 — 方案交付 | 7 | 3 项最终确认 → 写入 brainstorm.md → 文件验证 | 必须 |

**输出文件**：`$DOC_PATH/brainstorm.md`（含 YAML frontmatter）

**状态文件**：`$DOC_PATH/.brainstorm-state.json`

**项目名称提取**：支持 4 种 CASE（自然语言描述 / 明确结构 / 命令行参数 / 无法提取反问）

**约束**：
- 每个阶段必须暂停等待用户确认
- 技术选型必须给出 ≥2 种方案及 5 维对比
- 架构方案必须给出 2 种候选
- 只能修改 brainstorm.md 和状态文件

### 4.2 dev-flow（总调度）

| 属性 | 值 |
|------|-----|
| 模式 | primary agent |
| 温度 | 0.0 |
| 权限 | 读写 + bash（完整）+ webfetch |
| 关键配置 | bash: "*" allow |

**职责**：
1. 启动时初始化状态文件并校验完整性
2. 检查是否有 brainstorm.md 路径传入，作为 dev-plan 额外输入
3. 编排 dev-plan → 计划确认（≤5轮）→ dev-code（分批，含链折叠）→ dev-review → dev-bugfix（循环≤3轮）→ 交付
4. 管理修复循环计数（iteration），超限上报
5. 检测依赖图中的线性链（入度=1且出度=1），将链长 ≥ 2 的链折叠为一批，减少批次数
6. 依赖深度检查（>3层输出告警，不阻断）
7. 批次完成判定：grep plan.md 核验任务是否全部 [x]
8. 交付前提取文件清单并生成提交信息

**文件通信协议**：

| 步骤 | 子 agent | 输出文件 | 写入策略 | 额外参数（dev-flow 传入） | 输入文件（子 agent 自读） |
|------|---------|---------|---------|------------------------|---------------------|
| dev-plan（首次） | dev-plan | `plan.md` | 覆盖 | 原始需求 + brainstorm_path（如有） | — |
| dev-plan（修正） | dev-plan | `plan.md` | 覆盖 | 修改意见 | `plan.md` |
| dev-code（每批） | dev-code | `code.md` | 追加 | 当前批次号、已生成文件列表 | `plan.md` |
| dev-review | dev-review | `review.md` | 覆盖 | — | `code.md`、`plan.md` |
| dev-bugfix（每轮） | dev-bugfix | `bugfix.md` | 追加 | — | `review.md`、`plan.md` |

每次调用子 agent，必须传入 `$DOC_PATH`。输入文件由子 agent 自行读取。

**链折叠算法**：

```
1. 从依赖图中扫描"单进单出"线性链
   - 中间任务：入度=1 且 出度=1
2. 链长 ≥ 2 → 折叠为同一批次
   - 批次名格式：链首任务→链尾任务
3. 折叠后校验：批次内任务完成状态检查逻辑不变
4. 统计：原始 N 批 → 折叠后 M 批（缩减率 = (N-M)/N * 100%）
```

**依赖深度检查**：
- 折叠后总层级 ≤ 3 → 跳过
- 折叠后总层级 > 3 → 输出告警写入 code.md 头部（不阻断流程）

**约束**：
1. 严格按流程执行，不能以任何理由跳过
2. 计划确认是强制环节
3. 修复后必须重新审查
4. 修复循环最多 3 轮

### 4.3 dev-plan（规划）

| 属性 | 值 |
|------|-----|
| 模式 | subagent |
| 温度 | 0.0 |
| 权限 | 只读 + webfetch |
| 关键配置 | write: false, edit: false, bash: false |

**流程**：

| 步骤 | 内容 |
|------|------|
| 步骤0 | 上下文感知：检查 plan.md 是否存在 → 若存在则为修正模式，若不存在则为首次规划 |
| 步骤0a | 修正模式：读取已有计划，基于用户修改意见修正（非修改部分保持不变） |
| 步骤0b | 首次规划：若 prompt 含 brainstorm_path，读取 brainstorm.md 合并到需求中 |
| 步骤1 | 项目现状分析：识别项目结构、已有文件、语言/框架 |
| 步骤2 | 需求分析与规划：拆解功能→确定技术选型→加载编码规范→设计架构→任务拆分 |

**任务拆分规范**：

- 每个功能 → 数据模型 + 实现方案 + 约束边界 + 风险点 + 开发任务
- 任务标注：优先级、耗时、依赖关系、验收条件
- 验收条件格式：`{类型}: {具体值}`（类型：`file:` / `test:` / `api:` / `build:` / `cli:` / `schema:`）
- 禁止不可度量表述（如"功能正常""运行正确"）
- 单功能内任务依赖深度 ≤ 3 层（含跨功能依赖）
- 超出 3 层须 flatten 合并

**输出格式**（固定模板）：
1. 计划状态（已完成 / 待补充）
2. 项目现状（文件/目录、语言/框架）
3. 技术选型（语言/框架/版本）
4. 架构设计/文件结构
5. 需求分析与实现方案（按功能组织，含数据模型+方案+约束+风险+任务清单）
6. 全局风险与依赖（含版本号、全局排除项）
7. 已加载编码规范
8. plan.md 说明

### 4.4 dev-code（编码）

| 属性 | 值 |
|------|-----|
| 模式 | subagent |
| 温度 | 0.2 |
| 权限 | 读写 + bash（完整）+ webfetch |
| 关键配置 | write/edit: allow, bash: "*" allow |

**流程**：

| 步骤 | 内容 |
|------|------|
| 步骤1 | 读取 plan.md（语言/框架/编码规范/架构/当前批次任务/验收条件） |
| 步骤2 | 严格按计划编写代码，不随意添加未规划功能，遵循编码规范 |
| 步骤3 | 每完成一个文件立即语法自检（失败则修复后再继续） |
| 步骤4 | 依赖管理（按需安装新依赖） |

**批次完成更新**：
- 每完成一批 Level 任务后，必须更新 plan.md
- 将 `- [ ] 任务X` 替换为 `- [x] 任务X`（只修改复选框，保留任务名原文）
- 部分完成或跳过的任务不勾选，在报告中说明

**语法自检命令**：

| 语言 | 命令 |
|------|------|
| JS/TS | `node --check <file>` 或 `npx tsc --noEmit` |
| Python | `python -m py_compile <file>` |
| Go | `go vet` 或 `go build -o /dev/null ./...` |
| Rust | `cargo check` |
| C/C++ | `gcc -fsyntax-only <file>` 或 `msbuild /t:build` |

**输出格式**：
1. 编写状态（已完成/部分完成）
2. 实现功能 & 涉及文件
3. 新增依赖（如有）
4. 待补充内容
5. 任务完成度（已/未/跳过，任务名必须与 plan.md 一致）

### 4.5 dev-review（审查 — 质量门禁）

| 属性 | 值 |
|------|-----|
| 模式 | subagent |
| 温度 | 0.0 |
| 权限 | 只读 + bash（完整） |
| 关键配置 | write: false, edit: false, bash: "*" allow |

**流程**：

| 步骤 | 内容 | 依赖 Skill |
|------|------|-----------|
| 步骤1 | 读取 code.md + plan.md，提取语言/框架，加载编码规范 | *-coding-standards |
| 步骤2 | 多维度审查（架构→错误处理→安全→资源→质量→测试覆盖）+ 语言特定规范 | *-coding-standards |
| 步骤3 | 加载 build-verify，执行构建+类型+Linter + 运行测试 | build-verify |

**审查维度**（按优先级）：

| 优先级 | 维度 | 检查要点 |
|--------|------|---------|
| P0 | 架构设计 | 模块职责单一、耦合度合理、分层清晰 |
| P1 | 错误处理 | 异常捕获/传播、错误信息清晰、panic 风险 |
| P2 | 安全性 | SQL注入/XSS/CSRF、敏感信息硬编码、权限校验 |
| P3 | 资源管理 | 内存/连接池/文件句柄释放、资源泄漏 |
| P4 | 代码质量 | 命名规范、重复代码、魔法数字、函数长度 |
| P5 | 测试覆盖 | 边界条件、异常路径、核心逻辑测试 |

**问题分级**：

| 级别 | 标准 | ID 格式 |
|------|------|---------|
| Critical | 程序崩溃/数据丢失/资源泄漏 | C-001 |
| Major | 影响功能正确性或性能 | M-001 |
| Minor | 代码风格/可维护性问题 | m-001 |
| Potential | 潜在风险 | P-001 |

**阻塞规则**：构建和类型检查失败 → 记录为 Critical 级问题；Linter 问题记录但不强制阻塞。

**输出格式**：
1. 审查状态（通过/不通过）
2. 审查范围（语言/框架、文件数、审查类型、测试覆盖）
3. 代码优点
4. 问题清单（每个含 ID/级别/描述/位置/影响/代码对比/修复方案）
5. 验证清单（带 `- [ ]` / `- [x]` 标记）
6. 已加载编码规范

### 4.6 dev-bugfix（修复）

| 属性 | 值 |
|------|-----|
| 模式 | subagent |
| 温度 | 0.1 |
| 权限 | 读写 + bash（受限） |
| 关键配置 | edit: allow, bash: 白名单模式 |

**bash 权限白名单**：Node.js 构建测试 / Python 构建测试 / Go 构建测试 / C/C++ 构建 / Git 追溯 / Linter / 格式化

**流程**：

| 步骤 | 内容 |
|------|------|
| 步骤1 | 读取 review.md（问题清单+验证清单）+ plan.md（技术方案），加载编码规范 |
| 步骤1.5 | git log/blame 追溯近期变更 |
| 步骤2 | 问题定位与根因分析：运行复现→分析根因（逻辑/异常/并发/依赖/类型） |
| 步骤3 | 实施修复：最小化修改，不引入新功能，不重构无关代码 |
| 步骤4 | 编译自检（build-verify），失败则回退重试 |
| 步骤5 | 回归验证：确认原 BUG 已修复 |

**编译自检 vs 完整验证**：

| 维度 | dev-bugfix 步骤4 | dev-review 步骤3 |
|------|:---:|:---:|
| 构建验证 | ✅ | ✅ |
| 类型检查 | ❌ | ✅ |
| Linter | ❌ | ✅ |
| 代码审查 | ❌ | ✅ |
| 测试 | ❌ | ✅ |

**输出格式**：
1. 修复状态（已修复/部分修复/无法修复）
2. 根因分析（问题根源 + 定位方法）
3. 已修复问题（含修复说明）
4. 未修复问题（含原因）
5. 修改内容（含修复说明）
6. 验证结果（编译自检 + 回归验证）
7. 已加载编码规范
8. 修改文件列表

## 6. 错误处理与恢复

### 6.1 子 agent 调用重试

```
subagent 调用失败
      │
      ▼
记录错误到 errors.log
      │
      ▼
自动重试（最多 2 次，展示失败信息不等待确认）
   │
   ├── 成功 → 继续流程
   └── 2 次均失败 → 上报「多次重试失败，请人工介入」
                    → 展示涉及文件列表，提示手动回滚命令
```

- **重试上限**：每次 subagent 调用最多重试 2 次
- **重试不计数到 iteration**（重试和修复循环是两个独立维度）
- **子 agent 重新读取输入文件**，基于当前代码状态继续工作

### 6.2 方案确认循环

```
用户不满意计划
      │
      ▼
plan-revising（允许最多 5 轮）
      │
      ├── 轮数 < 5 → 收集修改意见 → 重新调用 dev-plan → 覆盖 plan.md → 回到确认
      └── 轮数 >= 5 → 终止流程，提示「请人工介入」
```

### 6.3 修复循环

```
dev-review 不通过
      │
      ▼
iteration += 1
      │
      ├── iteration < 3 → dev-bugfix → 回退到 dev-review 重新审查
      └── iteration >= 3 → status: "failed"
                          → 上报「多次修复未通过（已执行 3 轮），请人工介入」
                          → 保留工作区代码
```

- **修复循环上限**：3 轮
- **修复后必须重新审查**，不能直接交付

### 6.4 批次完成判定

```
dev-code 报告"已完成"
      │
      ▼
grep plan.md 核验本批所有任务
      │
      ├── 全部 [x] → 本批完成，启动下一批
      └── 仍有 [ ] → 本批未完成，同一批内重新调用 dev-code（不计 iteration，上限 2 次）
```

### 6.5 崩溃恢复

- **状态文件原子写入**：所有 JSON 状态文件采用 `write.tmp → mv` 模式
- **dev-flow 启动校验**：
  - 文件损坏 → 报错「状态文件损坏，请人工介入」
  - 文件存在且有效 → 检查 status：
    - `delivered` → 新流程
    - 其他 → 报错「上次流程未完成，请先清理目录后重试」
  - 文件不存在 → 新流程
- **dev-brainstorm 恢复**：读取 `.brainstorm-state.json`，按 `current-phase` 从中断阶段恢复

## 7. 状态文件生命周期

### 7.1 dev-flow 状态

```
初始化       → { iteration: 0, status: "planning" }
规划完成     → status: "planned"
计划确认通过 → status: "plan-confirmed"
计划要求修改 → status: "plan-revising"（回退到 planning）
计划用户终止 → status: "cancelled"
编码完成     → status: "coded"
审查完成     → status: "reviewed"
BUG 修复完成 → status: "bugfixed"（iteration += 1）
交付完成     → status: "delivered"
超限终止     → status: "failed"
```

### 7.2 dev-brainstorm 状态

```
初始化        → { status: "brainstorming", project: "...", theme: "...", current-phase: 1 }
阶段1 完成     → current-phase: 2
阶段2 完成     → current-phase: 3
阶段3 完成     → current-phase: 4
阶段4 完成     → current-phase: 5
阶段5 完成     → current-phase: 6
阶段6 完成     → current-phase: 7
方案交付完成   → status: "brainstorming-complete"
用户终止       → status: "cancelled"
```

所有写入均采用 tmp + rename 原子模式。

## 9. 关键约束

1. **计划确认是强制环节** — 未经用户确认不得进入编码阶段
2. **审查与修复回环必须执行** — 修复后必须重新审查，不能直接交付
3. **构建验证首次即执行** — 首次代码审查包含完整构建验证
4. **修复循环最多 3 轮** — 超限上报人工介入
5. **交付范围仅限代码文件** — 排除 `dev-flow/` 下的报告文件
6. **分批编码按依赖序（含链折叠）** — 前一批未完成不启动下一批；线性链自动折叠
7. **批次完成判定依赖 plan.md grep** — 机械化核验，不可凭主观判断
8. **编码规范按需加载** — 子 agent 从 plan.md 读取或自动探测语言后加载
9. **dev-brainstorm 入口门控强制** — 必须先执行请求分类
10. **dev-brainstorm 每阶段必须暂停确认** — 不得跳过任何确认环节
11. **提交信息仅保存不自动提交** — 由用户决定提交时机
12. **状态文件原子写入** — 所有 JSON 写入采用 tmp + rename 模式
