# dev-flow 开发流程设计文档

## 1. 概述

dev-flow 是一套基于 Master-Agent 模式的 AI 驱动全流程开发框架，覆盖从需求分析到代码交付的完整软件开发生命周期。

### 核心价值

- **全流程自动化**：需求 → 计划 → 编码 → 审查 → 修复 → 交付，减少人工介入
- **故障可恢复**：状态文件（原子写入）+ 文件变更追踪，支持 crash 后恢复
- **质量闭环**：构建验证 + 代码审查 → 自动修复循环 → 重新审查（最多 3 轮）
- **分层验证**：dev-review 做完整质量门禁（构建+类型+Linter+审查+测试），dev-bugfix 做轻量编译自检
- **文件即契约**：子 agent 之间通过文件通信，dev-master 不进行字段级解析

## 2. 架构设计

### 2.1 主从代理模式（Master-Agent）

```
                        ┌──────────────────┐
                        │   dev-master      │  ← Primary Agent（总调度）
                        │   流程编排/状态管理 │
                        └──┬───┬───┬───┬───┘
                           │   │   │   │
                    ┌──────┘   │   │   └──────────┐
                    ▼          ▼   ▼              ▼
            ┌──────────┐ ┌────────┐ ┌────────┐ ┌──────────┐
            │ dev-plan │ │dev-code│ │dev-review│ │dev-bugfix│
            │ 只读规划   │ │ 编码    │ │质量门禁   │ │ 修复     │
            │subagent  │ │subagent│ │subagent  │ │subagent  │
            └──────────┘ └────────┘ └──────────┘ └──────────┘
                         │
                         ▼
                  ┌──────────────┐
                  │git-autocommit│  ← 独立命令（自动提交）
                  └──────────────┘
```

### 2.2 组件角色矩阵

| 组件 | 模式 | 读写 | Bash | 温度 | 职责 |
|------|------|------|------|------|------|
| **dev-master** | primary | 读写 | 受限 | 0.0 | 流程总调度，状态管理，subagent 编排 |
| **dev-plan** | subagent | 只读 | 无 | 0.0 | 需求分析，技术选型，架构设计，任务拆分 |
| **dev-code** | subagent | 读写 | 受限 | 0.2 | 按计划编写代码，语法自检，依赖安装 |
| **dev-review** | subagent | 只读 | 完整 | 0.0 | 构建验证 + 类型检查 + Linter + 代码审查 + 测试 |
| **dev-bugfix** | subagent | 读写 | 完整 | 0.1 | BUG 定位与修复，格式化修复，编译自检 |

### 2.3 核心设计原则

1. **plan.md 是真理来源（SSOT）**：所有子 agent 从 plan.md 获取语言/框架、架构方案、验收条件
2. **文件即契约**：各子 agent 的输出格式在其定义文件中约定，dev-master 不进行字段级解析
3. **修改只影响一点**：如需增减字段，只需修改对应子 agent 的 agent 定义文件，不需要改 dev-master
4. **幂等性**：每个步骤可重复执行，状态文件确保流程连续性
5. **最小权限**：只读 agent（dev-plan, dev-review）没有 write/edit 权限
6. **人工兜底**：所有自动化路径都有上报人工介入的逃生口
7. **状态持久化**：JSON 状态文件 + git diff 文件追踪双重保障

## 3. 工作流详解

### 3.1 完整流程

```
用户输入 /coding-dev <功能名> [brainstorm.md路径]
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│ 初始化                                                    │
│  • 提取 $FEATURE_NAME                                     │
│  • 可选：读取 brainstorm.md，提取 key-decisions + risks    │
│  • 原子创建 .coding-dev-state.json（{iteration:0, status:"planning"}）│
│  • 校验状态文件 JSON 完整性                                 │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤1：探索需求                                          │
│  • 构建 prompt（brainstorm 上下文 / 原始用户需求）         │

                           │

┌──────────────────────────────────────────────────────────┐

│  • 展示摘要：技术选型 / 文件结构 / 任务清单 / 风险约束      │
│  • 询问用户：确认 / 修改 / 终止                           │
│  ┌────── 确认 ──► plan-confirmed                          │
│  ├────── 修改 ──► 回退 dev-plan 修正（≤5 轮）              │
│  └────── 终止 ──► cancelled                               │
└──────────────────────────┬───────────────────────────────┘
                           │

 │  • 按折叠后的批次顺序分批调用 @dev-code                    │

 │  • 前一批未完成不启动下一批                               │
 │  • 全量终检确认所有任务完成                               │
 │  • 更新状态：coded                                       │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤3：审查与BUG修复（闭环）                               │
│                                                          │
│ ┌─────────────────────────────────────────────┐          │
│ │ 3.1 审查代码                                 │          │
│ │  • 调用 @dev-review（只读）                    │          │
│ │  • 步骤：构建验证→类型检查→Linter→审查→测试    │          │
│ │  • 结果写入 review.md                          │          │
│ │  • 更新状态：reviewed                          │          │
│ └──────────────────┬──────────────────────────┘          │
│                    │                                      │
│               ┌────┴────┐                                │
│               ▼         ▼                                │
│          审查通过     审查不通过                           │
│               │         │                                │
│               │         ▼                                │
│               │  ┌──────────────────────────────┐        │
│               │  │ iteration >= 3?               │        │
│               │  │  ├── 否 → 3.2 调用 @dev-bugfix│        │
│               │  │  └── 是 → 上报人工介入，终止   │        │
│               │  └──────────────┬───────────────┘        │
│               │                 │                         │
│               │                 ▼                         │
│               │         ┌─────────────────────┐          │
│               │         │ 3.2 修复BUG           │          │
│               │         │  • 读取 review.md     │          │
│               │         │  • 格式化修复          │          │
│               │         │  • 复现→根因→修复      │          │
│               │         │  • 编译自检            │          │

│               │         │  • iteration += 1      │          │
│               │         │  • 结果写入 bugfix.md   │          │
│               │         │  • 回退到 3.1 重新审查  │          │
│               │         └─────────────────────┘          │
│               ▼                                           │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤4：交付成果                                          │
│  • 更新状态：delivering                                  │
│  • 从 code.md + bugfix.md 提取文件路径 → modified_files.txt│

└──────────────────────────────────────────────────────────┘
```

### 3.2 状态机

```
                    ┌──────────────────┐
                    │   planning (0)    │ ← 初始化
                    └────────┬─────────┘
                             │ dev-plan 完成
                             ▼
                    ┌──────────────────┐
                    │    planned        │
                    └────────┬─────────┘
                             │ 步骤1.2 用户确认
                             │
                  ┌──────────┼────────────┐
                  ▼          ▼            ▼
           ┌──────────┐ ┌──────────┐ ┌──────────┐
           │plan-conf.│ │plan-rev.│ │cancelled │
           │ 确认通过  │ │ 要求修改  │ │ 用户终止  │
           └────┬─────┘ └────┬─────┘ └──────────┘
                │            │
                │            └──→ 回退 planning
                ▼
           ┌──────────────────┐
           │ plan-confirmed   │
           └────────┬─────────┘
                    │ dev-code 完成
                    ▼
           ┌──────────────────┐
           │     coded         │
           └────────┬─────────┘
                    │ dev-review 完成
                    ▼
           ┌──────────────────┐
           │    reviewed       │
           └──┬───────────────┘
              │
    ┌─────────┴─────────┐
    ▼                   ▼
┌──────────────┐   ┌──────────────┐
│  delivering  │   │  bugfixed    │ ← iteration+1
└──────┬───────┘   └──────┬───────┘
       │                  │ 回退到 reviewed
       ▼                  ▼
┌──────────────┐   ┌──────────────┐
│  delivered   │   │  reviewed    │
└──────────────┘   └──────┬───────┘
                          │ iteration >= 3 ?
                          ├── 否 → bugfixed
                          └── 是 → failed
```

### 3.3 状态说明

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| `planning` | 初始化完成，规划中 | dev-master 启动 |
| `planned` | 规划完成，等待确认 | dev-plan 返回 |
| `plan-confirmed` | 用户确认计划 | 步骤1.2 用户确认 |
| `plan-revising` | 用户要求修改计划 | 步骤1.2 用户选择修改 |
| `cancelled` | 用户终止流程 | 步骤1.2 用户选择终止 |
| `coded` | 编码完成 | dev-code 完成 |
| `reviewed` | 审查完成 | dev-review 返回 |
| `bugfixed` | 修复完成（iteration+1） | dev-bugfix 返回 |
| `delivering` | 交付中 | 步骤4 开始 |
| `delivered` | 交付完成 | git-autocommit 完成 |
| `failed` | 超限终止（iteration≥3） | 修复循环超限 |

## 4. 各 Agent 详述

### 4.1 dev-master（总调度）

| 属性 | 值 |
|------|-----|
| 模式 | primary agent |
| 温度 | 0.0 |
| 权限 | 读写 + bash（受限）+ webfetch |
| 关键配置 | bash 仅允许 git status/rev-parse/checkout/commit + mkdir |

**职责**：
1. 启动时初始化状态文件并校验完整性
2. 可选读取 brainstorm.md 提取结构化上下文
3. 编排 dev-plan → 计划确认 → dev-code（分批，含链折叠优化）→ dev-review → dev-bugfix（循环）→ 交付
4. 管理修复循环计数（iteration），超限上报
5. 检测依赖图中的线性链（入度=1且出度=1），将链长 ≥ 2 的链折叠为一批，减少批次数
6. 交付前提取文件清单传递给 git-autocommit

**文件通信协议**：

| 步骤 | 输出文件 | 输入文件（供下步读取） |
|------|---------|---------------------|
| dev-plan（首次） | `plan.md` | — |
| dev-plan（修正） | `plan.md` | `plan.md`（前一次） |
| dev-code | `code.md` | `plan.md` |
| dev-review | `review.md` | `code.md` + `plan.md` |
| dev-bugfix | `bugfix.md` | `review.md` + `plan.md` |

**调用规范**：dev-master 仅传递功能名称、结果文件路径、必要轻量信息（如折叠后批次号），**禁止传递字段级上下文**。

### 4.2 dev-plan（规划）

| 属性 | 值 |
|------|-----|
| 模式 | subagent |
| 温度 | 0.0 |
| 权限 | 只读 + webfetch |
| 关键配置 | write: false, edit: false, bash: false |

**流程**：
1. **步骤0（修正模式）**：检查 plan.md 是否存在，若存在则读取已有计划 + 用户修改意见进行修正
2. **步骤1（首次规划）**：分析项目结构，扫描配置文件确定语言/框架，加载编码规范
3. **步骤2**：需求分析 → 技术选型 → 架构设计 → 任务拆分（含优先级/耗时/依赖/验收条件）

**输出格式**：固定模板，含计划状态、项目现状、技术选型、架构设计、功能实现方案（数据模型+方案+约束+风险+开发任务）、全局风险与依赖、已加载编码规范。

### 4.3 dev-code（编码）

| 属性 | 值 |
|------|-----|
| 模式 | subagent |
| 温度 | 0.2 |
| 权限 | 读写 + bash（受限）+ webfetch |
| 关键配置 | write/ed: allow, bash 允许目录操作和安装依赖 |

**流程**：
1. **步骤1**：从 plan.md 读取语言/框架、编码规范、架构方案、批次任务
2. **步骤2**：按计划编写代码，遵守编码规范，每完成一个文件记录路径
3. **步骤3**：每写完一个文件立即语法/编译自检（`node --check` / `py_compile` / `go vet` / `gcc -fsyntax-only`）
4. **步骤4**：安装计划中指定的新依赖

**自检要求**：自检失败则直接修复后再继续，不自检通过不交付。

### 4.4 dev-review（审查 — 质量门禁）

| 属性 | 值 |
|------|-----|
| 模式 | subagent |
| 温度 | 0.0 |
| 权限 | 只读 + bash（完整） |
| 关键配置 | write: false, edit: false, bash 允许所有构建/Linter/测试命令 |

**流程**（按执行顺序）：

| 步骤 | 内容 | 阻塞？ |
|------|------|--------|
| **步骤1** | 语言探测与上下文读取（code.md + plan.md） | — |
| **步骤2** | **构建验证**（参照 `auto-verify-code` 策略） | **是** |
| 2.1 | 优先级策略：项目脚本 → 工程脚本 → 通用命令 → 降级 | — |
| 2.2 | 环境检查：工具链缺失则降级 | — |
| 2.3 | 构建验证（各语言对应命令，退出码=0才通过） | **严重问题** |
| 2.4 | 类型检查（tsc --noEmit / mypy） | **严重问题** |
| 2.5 | Linter 检查（eslint / flake8 / go vet 等） | **不阻塞** |
| **步骤3** | 代码审查（规范、BUG、安全、性能、可维护性） | **是** |
| **步骤4** | 测试验证（运行单元测试） | **是** |
| **步骤5** | 可用性验证（必要时） | 可选 |

**构建命令表**：

| 语言 | 构建命令（按优先级） |
|------|-------------------|
| **JS/TS** | 1. `npm run build` 或 2. `npx tsc --noEmit` |
| **Go** | `go build ./...` |
| **Python** | `python -m py_compile <文件>` |
| **C/C++ (MSVC)** | `msbuild /t:rebuild` |
| **C/C++ (Qt)** | `qmake` + `jom` |
| **C/C++ (CMake)** | `cmake --build .` |
| **C/C++ (GCC)** | `gcc -o <output> <file>` 或 `g++ -o <output> <file>` |

### 4.5 dev-bugfix（修复）

| 属性 | 值 |
|------|-----|
| 模式 | subagent |
| 温度 | 0.1 |
| 权限 | 读写 + bash（完整） |
| 关键配置 | edit: allow, bash 允许构建/Linter/git 追溯/格式化 |

**流程**：
1. **步骤1**：从 review.md 读取问题清单 + 涉及文件 + 报错信息；git 追溯近期变更
2. **步骤1.5**：自动格式修复（prettier / black / gofmt / clang-format），纯格式问题修复后跳过步骤3~5
3. **步骤2**：复现问题，确认可稳定复现
4. **步骤3**：根因分析（逻辑错误/异常处理/并发/依赖变化/类型转换）
5. **步骤4**：最小化修复（只改必要代码，不引入新功能，不重构无关代码）
6. **步骤4.5**：**修复后编译自检**（快速确认不破坏编译，不执行完整类型检查/Linter）
7. **步骤5**：验证原 BUG 已修复 + 运行测试确认无回归

**编译自检 vs 完整验证**：

| 维度 | dev-bugfix 步骤4.5（修复自检） | dev-review 步骤2（质量门禁） |
|------|-------------------------------|---------------------------|
| 构建验证 | ✅ 快速编译确认 | ✅ 完整构建 |
| 类型检查 | ❌ 不执行 | ✅ 强制 |
| Linter | ❌ 不执行 | ✅ 记录但不阻塞 |
| 代码审查 | ❌ 不执行 | ✅ 完整审查 |
| 测试 | ❌ 不执行 | ✅ 单元测试 |
| 意图 | 快速反馈修复没破坏编译 | 全面的质量关卡 |

## 5. 数据流

### 5.1 上下文传递链

```
Step2: dev-master ──→ dev-code（按折叠后批次）
         │                           │
         │  plan.md 路径              │
         │  折叠后批次号              │
         │  （链批次名格式：链首→链尾）│
         │  前置依赖文件列表          │
         │                           ▼
         │                     输出 code.md
         │                           │
         ▼───────────────────────────┘
```

### 5.2 文件通信架构

```
coding-dev/$FEATURE_NAME/
├── .coding-dev-state.json    ← 全流程状态（dev-master 读写）
├── plan.md                   ← dev-plan 输出（所有子 agent 读取）
├── code.md                   ← dev-code 输出（dev-review 读取）
├── review.md                 ← dev-review 输出（dev-bugfix + dev-master 读取）
├── bugfix.md                 ← dev-bugfix 输出（dev-master 读取）
├── brainstorm-context.json   ← brainstorm 结构化数据（可选）
├── modified_files.txt        ← 交付文件清单
└── errors.log                ← 重试失败记录
```

### 5.3 文件变更追踪

```
subagent 调用前              subagent 调用后
      │                           │
      ▼                           ▼
git diff --name-only       git diff --name-only
      │                           │
      ▼                           ▼
（记录备用）               comm -13 before after
                                    │
                                    ▼
                             modified_files.txt  ← 用于重试回滚
```

## 6. 错误处理与恢复

### 6.1 重试机制

```
subagent 调用失败
      │
      ▼
记录错误到 errors.log
      │
      ▼
询问用户：是否重试（剩余 N 次）或终止
   ├── 是 → git checkout modified_files 回退 → 重试
   └── 否 → 终止流程
      │
      ▼
重试 2 次均失败 → 上报「多次重试失败，请人工介入」
```

- **重试上限**：每次 subagent 调用最多重试 2 次
- **重试前回滚**：从 code.md / bugfix.md 最后一段提取文件清单 → `git checkout -- <文件>`
- **重试不计数**到 `iteration`（重试和修复循环是两个独立维度）
- **超时阈值**：无上限（公司模型速度可能较慢）

### 6.2 修复循环

```
dev-review 不通过
      │
      ▼
iteration += 1
      │
      ├── iteration < 3 → dev-bugfix → 回退到 review
      │
      └── iteration >= 3 → 更新状态 failed
                          → 上报「多次修复未通过，请人工介入」
                          → 保留工作区现场代码供参考
```

- **修复循环上限**：3 轮
- **修复后必须重新审查**，不能直接交付
- **超限后保留工作区代码**，供人工现场排查

### 6.3 崩溃恢复

- **状态文件原子写入**：所有 `.coding-dev-state.json` 的写入采用 `write.tmp → mv` 模式，防止 crash 后半写文件
- **启动时校验**：dev-master 检查状态文件 JSON 完整性，损坏则报错，不存在则初始化
- **状态冲突检测**：若上次流程未完成（status ≠ delivered），报错并提示清理目录

## 9. 关键约束

1. **计划确认是强制环节** — 未经用户确认不得进入编码阶段
2. **审查与修复回环必须执行** — 修复后必须重新审查，不能直接交付
3. **构建验证首次即执行** — 首次代码审查也包含构建验证（不再跳过）
4. **修复循环最多 3 轮** — 超限上报人工介入
5. **交付范围仅限代码文件** — 排除 `coding-dev/` 下的报告文件
6. **分批编码按依赖序（含链折叠）** — 前一批未完成不启动下一批；线性链（单进单出）自动折叠为一批以减少轮次
7. **编码规范按需加载** — 子 agent 从 plan.md 读取或自动探测语言后加载对应规范
