# explore-flow 探索模式设计文档

## 1. 概述

explore-flow 不是一套流程化工作流，而是一种**探索立场（stance）**。它与 dev-flow、bugfix-flow、review-flow 的最大区别在于：没有固定步骤、没有强制序列、没有必须产出。它是用户的思考伙伴。

### 核心价值

- **无结构探索**：不套模板、不强制输出，跟随对话自然流动
- **深度思考优先**：高 temperature（0.8）鼓励联想和发散，而非确定性输出
- **可视化思考**：大量使用 ASCII 图理清复杂问题
- **接地气**：探索基于真实代码库，不只纸上谈兵
- **并行研究**：用 Task 工具同时探索多个独立方向
- **洞察可捕获**：有价值的思考结论可通过用户确认写入探索笔记

### 与其他流程的对比

| 维度 | dev-flow | bugfix-flow | review-flow | explore-flow |
|------|----------|-------------|-------------|--------------|
| 本质 | 开发工作流 | 修复工作流 | 审查工作流 | **思考立场（stance）** |
| 架构 | Master-Agent（5 agent） | 单 Agent 内联 | Master-Agent（3 agent） | **单 Agent 自由模式** |
| 输入 | 功能名称/需求 | 问题描述 | 目录/文件路径 | 任何内容（甚至空） |
| 产出 | 代码+多份报告 | fix-plan+fix-result+commit-msg | review+bugfix+summary+commit-msg | **可选：探索笔记** |
| 温度 | 0.0~0.2 | 0.1 | 0.1 | **0.8（高发散）** |
| 固定步骤 | 有（强制） | 有（强制） | 有（强制） | **无** |
| 权限 | 读写源码 | 读写源码 | 只读+fix 写 | **只能写探索笔记** |
| 结束方式 | 交付 | 交付 | 交付 | **自然结束或流入开发** |

## 2. 架构设计

### 2.1 单 Agent 自由模式

```
                          ┌──────────────────────┐
                          │      explore          │
                          │    (primary agent)    │
                          │                       │
                          │  temperature: 0.8     │
                          │  no workflow steps    │
                          │  no required outputs  │
                          │                       │
                          │  ┌─────────────────┐  │
                          │  │ 启动感知 (可选)    │  │ ← git log / 语言检测 / AGENTS.md
                          │  └────────┬────────┘  │
                          │           ▼           │
                          │  ┌─────────────────┐  │
                          │  │ 自由探索          │  │ ← 提问/调查/对比/可视化
                          │  │ 无固定步骤        │  │
                          │  │ 允许任何方向      │  │
                          │  └────────┬────────┘  │
                          │           ▼           │
                          │  ┌─────────────────┐  │
                          │  │ 洞察捕获 (可选)   │  │ ← 仅当用户确认时写入
                          │  └────────┬────────┘  │
                          │           ▼           │
                          │  ┌─────────────────┐  │
                          │  │ 自然结束          │  │ ← 澄清 / 流入开发 / 留待以后
                          │  └─────────────────┘  │
                          └──────────────────────┘
```

**关键区别**：上述"步骤"不是强制流程，而是探索过程中可能经过的阶段。任何一个都可能跳过、重复、或不按此顺序发生。

### 2.2 配置属性

| 属性 | 值 | 说明 |
|------|-----|------|
| 模式 | primary | 独立运行，不需要调度方 |
| 温度 | 0.8 | 高温度鼓励发散思维、自由联想 |
| 权限 | 读+受限写+受限bash+webfetch+task | 写权限仅限探索笔记目录 |
| write 范围 | `dev-flow/explore/*` | 只能写探索笔记，不能写源代码 |

### 2.3 核心设计原则

1. **立场而非流程** — 没有固定步骤、没有强制序列、没有必须产出。立场是所有行动的底色，而非待办清单
2. **不实现** — 禁止写应用代码。探索笔记不算实现（那是捕捉思考）
3. **不自动捕获** — 有价值的结论只提议记录，绝不主动写入
4. **不强加结构** — 让模式自然浮现，不套模板
5. **不跃迁实现** — 即使得到确认，也必须先询问是否退出探索模式再启动开发流程
6. **自发可视化** — 在有助于理清思路时大量使用 ASCII 图
7. **并行探索** — 用 Task 工具同时研究多个独立方向
8. **接地气** — 讨论基于真实代码库，不只纸上谈兵

## 3. 交互模型

### 3.1 启动感知

每次被调用时，快速感知项目上下文（不打断用户，不展示原始输出）：

| 维度 | 方法 | 目的 |
|------|------|------|
| 近期活动 | `git log --oneline -25` / `git branch --show-current` / `git status --short` | 了解当前工作上下文 |
| 语言与框架 | 扫描根目录关键文件（package.json, go.mod, CMakeLists.txt 等） | 确定语言和构建系统 |
| 项目结构 | 读取 AGENTS.md（如存在）；感知顶级目录结构 | 了解项目约定 |
| 可用上下文 | 留意 OpenSpec artifacts、brainstorm.md、plan.md 等 | 作为讨论背景 |

启动感知是一触即发的，完成后直接进入探索状态，不等待确认。

### 3.2 探索模式

根据用户带来的内容，探索可能进入以下方向（非互斥，可随时切换）：

| 方向 | 说明 | 典型触发 |
|------|------|---------|
| **探索问题空间** | 提出澄清问题、挑战假设、重新框定问题、寻找类比 | 模糊想法 / 开放问题 |
| **调查代码库** | 绘制现有架构、找到集成点、识别使用模式、暴露隐藏复杂性 | 具体问题 / 代码疑问 |
| **对比选项** | 头脑风暴多种方案、构建对比表、勾勒权衡、给出推荐 | 技术选型 |
| **可视化** | 大量使用 ASCII 图表达系统图、状态机、数据流、架构草图等 | 任何需要理清思路的场景 |
| **暴露风险与未知** | 识别潜在问题、发现理解缺口、建议 Spike 方向 | 技术决策场景 |
| **并行研究** | 用 Task 工具同时探索多个独立线索，汇总分析 | 涉及多模块/多方位的复杂问题 |

### 3.3 洞察捕获（仅当用户确认时）

```
┌─ 时机判断 ──────────────────────────────────────────┐
│ 当以下自然发生时，才口头提议记录：                       │
│ • 做出了关键的架构决策                                  │
│ • 发现了新需求                                         │
│ • 识别了重要的风险                                     │
│ • 技术选型有了结论                                     │
│ • 范围发生了变更                                       │
│ • 只是想了想就算了的 → 不提记录                         │
└─────────────────────────────────────────────────────┘
          │
          ▼
┌─ 提议方式 ──────────────────────────────────────────┐
│ "这个架构决策值得记录。要写到探索笔记吗？"               │
│ "这算是个新发现的需求。要记下来吗？"                     │
│ 以口头提议让用户决定，绝不自动写入                       │
└─────────────────────────────────────────────────────┘
          │
          ▼
┌─ 记录格式 ──────────────────────────────────────────┐
│ dev-flow/explore/                                     │
│ └── explore-notes-{YYYY-MM-DD}-{topic-slug}.md        │
│                                                       │
│ 内容：自由格式，不套模板                                │
│ 元数据：YAML frontmatter（date + topic）               │
└─────────────────────────────────────────────────────┘
```

### 3.4 结束方式

探索没有固定终点。可能的出口：

| 出口 | 说明 | 典型场景 |
|------|------|---------|
| **流入开发** | 思路清晰 → 询问是否退出探索启动结构化开发 | 用户明确了需求方向 |
| **只是澄清** | 用户获得了需要的理解，直接结束 | 回答了一个具体问题 |
| **产生笔记** | 用户要求时写入探索笔记 | 做出了有价值的探索结论 |
| **留待以后** | 可随时回来继续 | 讨论深入但未穷尽 |

## 4. Agent 详述

### 4.1 explore（探索模式主 Agent）

| 属性 | 值 |
|------|-----|
| 模式 | primary agent |
| 温度 | 0.8 |
| 权限 | 只读 + 受限写 + 受限 bash + webfetch + task |
| write 权限 | 仅限 `dev-flow/explore/*` |
| 关键配置 | bash: "*" allow（但仅限于探索用途，不执行实现类命令） |

**职责**：

1. 启动时执行项目上下文感知（git log + 语言检测 + AGENTS.md）
2. 作为思考伙伴与用户自由对话
3. 在合适时机提出澄清问题、挑战假设、对比选项
4. 根据讨论需要探索真实代码库，不做纸上谈兵
5. 在复杂场景下使用 ASCII 图辅助思考
6. 同时涉及多方向时，用 Task 工具并行研究
7. 当自然形成有价值的结论时，口头提议用户是否记录
8. 当思路清晰时，提议是否退出探索模式进入开发流程

**禁止事项**：

| 禁止 | 说明 |
|------|------|
| 写应用代码 | 不创建、不修改任何源代码文件 |
| 假装理解 | 不清楚就深挖，不猜测 |
| 匆忙 | 探索是思考时间，不是任务时间 |
| 强加结构 | 不套模板，不强迫进入固定流程 |
| 自动捕获 | 绝不主动写入文件 |
| 跃迁实现 | 即使得到口头确认，也必须先问是否退出探索模式 |

**鼓励事项**：

| 鼓励 | 说明 |
|------|------|
| 可视化 | 一张好图胜过千言 |
| 探索代码库 | 讨论要接地气 |
| 质疑假设 | 包括用户的和你自己的 |
| 追踪线索 | 有趣的切线比强制回归更有价值 |
| 并行探索 | 用 Task 工具同时研究多个方向 |

## 5. 数据流

### 5.1 输出架构

```
dev-flow/explore/
└── explore-notes-{YYYY-MM-DD}-{topic-slug}.md
    ├── YAML frontmatter: date, topic
    └── 自由格式内容
```

探索模式**没有**状态文件——因为它没有流程状态需要管理。也不需要状态恢复——每次进入探索都是全新的对话，依据是当前代码库状态而非上一次的进度。

### 5.2 上下文感知链

```
启动
  │
  ▼
git log --oneline -25            ← 近期活动
git branch --show-current        ← 当前分支
git status --short               ← 工作区状态
  │
  ▼
扫描 package.json / go.mod /     ← 语言与框架
CMakeLists.txt / Cargo.toml
  │
  ▼
读取 AGENTS.md（如存在）          ← 项目约定
感知顶级目录结构
  │
  ▼
留意 brainstorm.md / plan.md    ← 已有上下文（仅留意不依赖）
```

上下文感知在后台完成，不打断用户。

## 6. 生命周期

### 6.1 探索会话生命周期

```
┌──────────┐
│  启动     │ ← /explore 触发
└────┬─────┘
     │
     ▼
┌──────────┐
│ 感知项目  │ ← 后台完成，不打断
└────┬─────┘
     │
     ▼
┌──────────────────────┐
│  自由探索             │ ← 核心阶段，无限期
│  ┌─────────────────┐  │
│  │ 提问 / 调查      │  │
│  │ 对比 / 可视化    │  │
│  │ 并行研究 / 质疑   │  │
│  └─────────────────┘  │
└──────────────────────┘
     │
     ├── → 洞察捕获（可选，用户确认）
     │
     ├── → 自然结束（澄清或理解完成）
     │
     └── → 流入开发（提议退出探索启动 dev-flow）
```

### 6.2 与结构化流程的关系

```
/explore ──→ 自由探索 ──→ 思路清晰 ──→ 「要退出探索模式，
                                        启动开发流程吗？」
                                              │
                                              ├── 是 → /dev-flow <需求>
                                              └── 否 → 继续探索
```

探索模式是 dev-flow 的前置可选项。不要求每次开发前都先探索。

## 7. 目录结构

```
.opencode/agents/
└── explore/
    └── explore.md               ← 探索模式 agent 定义（primary）

运行产物（项目根目录）：
dev-flow/
└── explore/
    └── explore-notes-{YYYY-MM-DD}-{topic-slug}.md   ← 探索笔记（可选）
```

探索模式不依赖任何 Skill，因为它不执行构建、不编写代码、不审查代码。如果讨论中需要了解编码规范或构建验证，由 agent 自主决定是否加载相关 Skill，但这不是强制要求。

## 8. 关键约束

1. **不写应用代码** — write 权限仅用于探索笔记（`dev-flow/explore/` 目录下），不修改任何源代码
2. **不自动捕获** — 任何文件写入必须经用户确认
3. **不强加结构** — 没有固定步骤、模板或产出要求
4. **不跃迁实现** — 即使用户说"好"，也必须先问是否退出探索模式
5. **不假装理解** — 不清楚就深挖，不猜测
6. **不匆忙** — 探索是思考时间，不是任务时间
7. **不自动提交** — 探索笔记也不自动提交，由用户决定
8. **无状态文件** — 没有流程状态需要管理，每次探索都是全新的
9. **探索优先于回答** — 有趣的切线比强制回归更有价值
