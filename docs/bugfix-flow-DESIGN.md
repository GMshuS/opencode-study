# bugfix-flow Bug 修复流程设计文档

## 1. 概述

bugfix-flow 是一套独立的、内联式的 AI 驱动 Bug 修复流程框架，覆盖从根因分析到交付总结的完整闭环。它不依赖外部 subagent，所有逻辑在一个 primary agent 内完成。

### 核心价值

- **全内联架构**：单一 primary agent 完成全部流程，无 subagent 调度开销
- **方案驱动修复**：先分析根因 → 用户确认修复方案 → 再执行修复，避免盲目修改
- **质量闭环**：修复后必经 build-verify 编译验证（最多重试 2 次）才能交付
- **故障可恢复**：状态文件贯穿全流程，支持 crash 后从中断点恢复
- **人工可控**：方案确认强制用户决策，编译失败重试需用户确认
- **可追溯交付**：输出 fix-plan.md + fix-result.md + commit-msg.txt 三件套

### 与 review-flow / dev-flow 的对比

| 维度 | dev-flow | review-flow | bugfix-flow |
|------|----------|-------------|-------------|
| 场景 | 新功能开发 | 已有代码全面审查 | 针对具体 Bug 修复 |
| 架构 | Master-Agent（5 个 agent） | Master-Agent（3 个 agent） | 单 Agent 内联 |
| 输入 | 功能名称 / 需求描述 | 目录/文件路径 | 问题描述（报错/异常/复现步骤等） |
| 产出 | plan + code + review + bugfix | review + bugfix + summary + commit-msg | fix-plan + fix-result + commit-msg |
| 是否有审查环节 | 有（dev-review） | 有（review-flow-review） | 无（内联分析 + build-verify） |
| 修复循环 | review→fix 回环 3 轮 | fix→核验 3 轮 | 编译验证重试 2 次 |
| 方案确认循环 | 5 轮 | 5 轮 | 5 轮 |

## 2. 架构设计

### 2.1 单 Agent 内联架构

```
                         ┌──────────────────────┐
                         │    bugfix-flow        │
                         │    (primary agent)    │
                         │                       │
                         │  ┌─────────────────┐  │
                         │  │ 步骤1: 分析+确认  │  │
                         │  │ (≤5轮, 含根因分析) │  │
                         │  └────────┬────────┘  │
                         │           ▼           │
                         │  ┌─────────────────┐  │
                         │  │ 步骤2: 执行修复   │  │
                         │  │ (按 fix-plan.md)  │  │
                         │  └────────┬────────┘  │
                         │           ▼           │
                         │  ┌─────────────────┐  │
                         │  │ 步骤3: 编译验证   │  │
                         │  │ (≤2次重试)       │  │
                         │  └────────┬────────┘  │
                         │           ▼           │
                         │  ┌─────────────────┐  │
                         │  │ 步骤4: 交付文档   │  │
                         │  └─────────────────┘  │
                         └──────────────────────┘
                                  │
                    ┌─────────────┴─────────────┐
                    ▼                           ▼
            ┌──────────────┐           ┌──────────────┐
            │ build-verify │           │git-autocommit │
            │   (skill)     │           │  (独立命令)    │
            └──────────────┘           └──────────────┘
```

### 2.2 配置属性

| 属性 | 值 | 说明 |
|------|-----|------|
| 模式 | primary | 独立运行，不需要调度方 |
| 温度 | 0.1 | 低温度保证一致性 |
| 权限 | 读写 + bash（完整）+ webfetch | 需要修改源码和运行构建命令 |
| 依赖 Skill | build-verify | 步骤3 编译验证 |

### 2.3 核心设计原则

1. **内联优先**：所有逻辑在主 agent 内完成，避免 subagent 调度的通信和上下文传递开销
2. **先方案后修复**：必须先输出根因分析 + 修复方案并获用户确认，再执行代码修改
3. **修复方案持久化**：确认后的方案写入 fix-plan.md，作为步骤2 的权威指导
4. **逐步交付**：每个步骤有独立的产物文件，各步骤之间通过状态文件衔接
5. **人工兜底**：方案确认循环、编译验证重试均需用户确认，所有异常路径有上报人工介入的逃生口
6. **不自动提交**：提交信息保存到 commit-msg.txt，由用户决定提交时机

## 3. 工作流详解

### 3.1 完整流程

```
用户输入问题描述（报错信息 / 异常输出 / 复现步骤 / 预期行为等）
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│ 初始化                                                    │
│  • 从用户输入提取问题描述                                   │
│  • 生成 $BUGFIX_ID（如 login500failed）                    │
│  • 获取日期 $DATE（YYYYMMDD）                              │
│  • 设置 $DOC_PATH = ./bugfix-flow/$DATE/bugfix-$BUGFIX_ID │
│  • 创建输出目录                                            │
│  • 状态恢复校验（见 6.4 节）                                │
│  • 首次初始化：.flow-state.json = {status:"analyzing", problem:"..."}│
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤1：问题分析与方案确认（≤ 5 轮）                          │
│  • $ROUND_COUNT = 1                                      │
│                                                          │
│  ┌── 第1轮：完整分析 ───────────────────────────────────┐  │
│  │ • 阅读问题描述 + 关联代码，理解业务逻辑                  │  │
│  │ • 定位根因（运行复现/代码推演/静态分析/配置检查等）       │  │
│  │ • 输出修复方案：根因分析 + 分析过程 + 修改点列表 + 影响范围│  │
│  │ • 更新状态：analyzed                                   │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌── 第2~5轮：定向补充分析 ─────────────────────────────┐  │
│  │ • 根据用户上轮意见定向补充                              │  │
│  │ • 缩小范围 / 替代方案 / 新线索重新分析                   │  │
│  │ • 更新状态：analyzed                                   │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                          │
│  每轮结束询问用户：确认执行 / 修改方案 / 终止                 │
│    ├── 确认执行 → fix-plan.md + status:"confirmed"        │
│    ├── 修改方案 → $ROUND_COUNT += 1，下一轮（≤5）          │
│    └── 终止 → status:"cancelled"，终止                    │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤2：执行修复                                           │
│  • 初始化 $MODIFIED_FILES = []                            │
│  • 按 fix-plan.md 逐个文件修改代码                          │
│  • 每次修改后将文件路径追加到 $MODIFIED_FILES（自动去重）     │
│  • 更新状态：fixing                                        │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤3：编译验证（≤ 2 次重试）                               │
│  • 复用 build-verify skill（构建 + 类型 + Linter）          │
│  • 首次失败 → 记录 errors.log → 分析原因 → 修正代码         │
│  • 重试前询问用户「是否重试（剩余 N 次）或终止」              │
│  • 重试 2 次均失败 → 上报「请人工介入」                      │
│  • 编译通过 → 进入步骤4                                    │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤4：生成交付文档                                        │
│  • 生成 4 段式提交信息（问题来源/原因/修改说明/测试建议）     │
│  • 写入 commit-msg.txt（含提交信息 + 修改文件列表）          │
│  • 写入 fix-result.md（修复摘要 + 修改内容 + 涉及文件 + 验证）│
│  • 更新状态：delivered                                     │
│  • 终端展示精简摘要，提示可用 @git-autocommit 提交           │
└──────────────────────────────────────────────────────────┘
```

### 3.2 状态机

```
                  ┌──────────────────┐
                  │   analyzing       │ ← 初始化
                  └────────┬─────────┘
                           │ 步骤1 分析完成
                           ▼
                  ┌──────────────────┐
                  │   analyzed        │
                  └────────┬─────────┘
                           │ 步骤1 用户确认
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
   ┌──────────────┐ ┌──────────────┐ ┌──────────┐
   │  confirmed   │ │  analyzed    │ │ cancelled│
   │  用户确认执行  │ │ 用户要求修改   │ │ 用户终止  │
   └──────┬───────┘ └──────┬───────┘ └──────────┘
          │                │
          │                └──→ $ROUND_COUNT += 1（≤5轮）
          │
          ▼
   ┌──────────────────┐
   │    fixing         │ ← 步骤2 执行修复
   └────────┬─────────┘
            │ 步骤3 编译验证
            ├── 通过 → delivered
            └── 失败 → 修正 + 重试（≤2次）
                       └── 2次均失败 → 终止 + 上报人工
```

### 3.3 状态说明

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| `analyzing` | 初始化完成，开始分析 | bugfix-flow 启动 |
| `analyzed` | 分析完成，等待方案确认 | 步骤1 每轮分析结束 |
| `confirmed` | 用户确认修复方案 | 步骤1 用户选择确认执行 |
| `cancelled` | 用户终止流程 | 步骤1 用户选择终止 |
| `fixing` | 执行修复中 | 步骤2 开始执行修复 |
| `delivered` | 交付完成 | 步骤4 完成 |

## 4. 数据流

### 4.1 文件通信架构

```
bugfix-flow/$DATE/bugfix-$BUGFIX_ID/
├── .flow-state.json    ← 流程状态（全程读写）
├── fix-plan.md         ← 修复方案（步骤1 确认后写入，步骤2 读取）
├── fix-result.md       ← 修复结果报告（步骤4 写入）
├── commit-msg.txt      ← 提交信息 + 修改文件列表（步骤4 写入）
└── errors.log          ← 编译失败错误记录（步骤3 首次失败写入）
```

### 4.2 内存变量

| 变量 | 类型 | 作用域 | 说明 |
|------|------|--------|------|
| `$BUGFIX_ID` | string | 全流程 | 问题标识符，如 `login500failed` |
| `$DATE` | string | 全流程 | 日期，格式 `YYYYMMDD` |
| `$DOC_PATH` | string | 全流程 | 输出目录路径 |
| `$ROUND_COUNT` | int | 步骤1 | 方案确认轮次计数（1~5），不写入状态文件 |
| `$MODIFIED_FILES` | array | 步骤2~4 | 修改文件路径列表，自动去重 |

### 4.3 状态文件格式

```json
{
  "status": "analyzing",
  "problem": "问题描述文本"
}
```

- `status`：当前流程状态
- `problem`：原始问题描述（初始化时写入，后续不变）

状态文件采用直接写入方式（非原子模式），因为 bugfix-flow 为单 agent 内联执行，不存在并发写入风险。

## 5. 各步骤详述

### 5.1 步骤1：问题分析与方案确认

**输入**：用户提供的问题描述（报错信息、异常输出、复现步骤、预期行为等）

**分析策略**（根据问题类型灵活选择）：

| 策略 | 适用场景 | 方法 |
|------|---------|------|
| 运行复现 | 可复现的运行时错误 | 运行命令重现问题，获取堆栈/错误输出 |
| 代码推演 | 逻辑错误 | 追踪调用链，推演数据流和状态变化 |
| 静态分析 | 语法/类型/配置问题 | 分析代码结构、配置项、依赖关系 |
| 配置检查 | 环境/配置文件错误 | 检查配置项格式、路径、环境变量 |
| 边界条件 | 输入异常/空值/溢出 | 分析边界输入的处理逻辑 |
| 异常处理 | 未捕获/错误传播 | 检查异常处理的完整性 |
| 并发排查 | 多线程/协程问题 | 分析锁、竞争条件、死锁 |
| 类型推断 | 类型不匹配 | 推断变量类型流转 |

**输出格式**（修复方案）：

```markdown
# 修复方案

## 根因分析
[精简的根因说明]

## 问题分析过程
[根因定位过程：代码调用链分析、复现步骤、逻辑推演、静态分析等]

## 修改点列表
- **[文件 1]**：[行号范围] - [修改说明]
  ```lang
  // 修改前
  ...
  // 修改后
  ...
  ```
- **[文件 2]**：[行号范围] - [修改说明]
  ...

## 影响范围
[影响评估]
```

**确认交互**：每轮结束提问「请确认（第 N/5 轮）：『确认执行 / 修改方案 / 终止』」

- **确认执行**：写入 fix-plan.md，更新状态 `confirmed`
- **修改方案**：`$ROUND_COUNT += 1`，补充分析
  - 缩小范围 → 聚焦特定模块深入
  - 方案不可行 → 分析原因，提出替代方案
  - 新线索 → 结合新线索重新分析
- **终止**：更新状态 `cancelled`

**循环上限**：5 轮，超限终止并上报「方案多次未通过确认，请人工介入」

### 5.2 步骤2：执行修复

**输入**：fix-plan.md（已确认的修复方案）

**执行规则**：
1. 按 fix-plan.md 中修改点列表逐项修改代码
2. 每次修改文件后，将文件路径追加到 `$MODIFIED_FILES`（自动去重）
3. 只改动方案中指定的必要代码，不重构无关代码
4. 更新状态 `fixing`

### 5.3 步骤3：编译验证

**输入**：修改后的代码 + build-verify skill

**执行规则**：
1. 加载 build-verify skill，执行完整验证流程（构建 + 类型检查 + Linter）
2. **首次失败**：
   - 记录错误到 `$DOC_PATH/errors.log`
   - 分析编译错误原因，修正代码
   - 修正涉及的文件追加到 `$MODIFIED_FILES`（去重）
3. **重试**：询问用户「是否重试（剩余 N 次）或终止」
4. **重试 2 次均失败** → 上报「多次重试失败，请人工介入」
5. **编译通过** → 进入步骤4

**重试上限**：2 次，与方案确认循环互不计数。

### 5.4 步骤4：生成交付文档

**产物 1 — commit-msg.txt**：4 段式提交信息 + 修改文件列表

```
问题来源：【问题来源描述】
问题原因：【根因说明】
修改说明：【修改方法及修改点】
测试建议：【对修改点的测试建议】
---
修改文件列表：
文件A
文件B
```

**产物 2 — fix-result.md**：Bug 修复结果报告

```markdown
# Bug 修复结果

## 修复摘要
- **问题**：[问题描述]
- **根因**：[根因摘要]
- **状态**：【已修复 / 部分修复 / 无法修复】

## 修改内容
[修复代码说明 / 修改行数]

## 涉及文件
[文件列表]

## 构建验证结果
- 构建：【通过/失败】命令：[xxx]
- 类型检查：【通过/失败/跳过】
- Linter：【通过/发现问题】

## 提交信息
详见 `commit-msg.txt`
```

**终端展示摘要**：

```
────────────────────────────────────────
 Bug 修复完成：bugfix-$BUGFIX_ID

 问题：[问题描述]
 根因：[根因摘要]
 修改：[文件数] 个文件

 报告文件:
   fix-plan.md      - 修复方案
   fix-result.md    - 修复结果（含验证记录）
   commit-msg.txt   - 提交信息 + 修改文件列表

 执行 @git-autocommit $DOC_PATH/commit-msg.txt 提交
────────────────────────────────────────
```

## 6. 错误处理与恢复

### 6.1 方案确认循环

```
$ROUND_COUNT = 1
     │
     ▼
完整分析 → 展示修复方案 → 用户选择
     │                        │
     ├── 确认执行 → confirmed   │
     ├── 终止 → cancelled       │
     └── 修改 → $ROUND_COUNT += 1
                    │
                    ▼
              $ROUND_COUNT > 5 ?
              ├── 否 → 定向补充分析 → 回到展示
              └── 是 → 终止 + 上报人工
```

### 6.2 编译验证重试

```
编译验证
   │
   ├── 通过 → 进入步骤4
   └── 失败
        │
        ▼
   首次失败 → 记录 errors.log → 分析修正 → 追加 $MODIFIED_FILES
        │
        ▼
   询问用户：是否重试（剩余 N 次）或终止
        ├── 是 → 重新编译验证
        └── 否 → 终止
        │
        ▼
   重试 2 次均失败 → 终止 + 上报人工
```

### 6.3 重试策略总结

| 维度 | 上限 | 用户确认 | 触发条件 |
|------|------|---------|---------|
| 方案确认 | 5 轮 | 每轮确认 | 用户选择修改 |
| 编译验证 | 2 次重试 | 每次重试前确认 | 编译失败 |

两个维度互不计数，独立管理。

### 6.4 崩溃恢复

启动时读取 `.flow-state.json`：

- **文件存在且 JSON 完整** → 校验后按当前 `status` 恢复：
  - `analyzing` / `analyzed` → 从步骤1 恢复
  - `confirmed` → 校验 fix-plan.md 存在，从步骤2 恢复
  - `fixing` → 从步骤3 恢复（重新编译验证）
  - `delivered` → 提示已完成，按新流程执行
  - `cancelled` → 提示已被取消
- **文件损坏无法解析** → 报错「状态文件损坏，请人工介入」
- **文件不存在** → 进入首次初始化

## 7. 状态文件生命周期

```
初始化    → { status: "analyzing",  problem: "..." }
分析完成  → { status: "analyzed" }
方案确认  → { status: "confirmed" }
方案终止  → { status: "cancelled" }
执行修复  → { status: "fixing" }
交付完成  → { status: "delivered" }
```

每次状态变化同步更新。

## 9. 关键约束

1. **方案确认是强制环节** — 未经用户确认不得进入修复执行
2. **先方案后修复** — 必须先输出根因分析 + 修改点列表，获确认后才能修改代码
3. **分析与确认循环最多 5 轮** — 超限终止并上报人工介入
4. **编译验证最多重试 2 次** — 超限上报人工介入
5. **修复方案持久化** — 确认后写入 fix-plan.md，作为修复的权威指导
6. **不自动提交代码** — 提交信息仅保存到 commit-msg.txt，由用户决定提交时机
7. **修改文件追踪** — $MODIFIED_FILES 全程维护，自动去重
8. **状态文件同步更新** — 每次状态变化必须更新 .flow-state.json
9. **内联执行** — 不依赖外部 subagent，所有逻辑在 primary agent 内完成
10. **最小化修改** — 只改方案中指定的必要代码，不重构无关部分
